# Phase 1 — Backend foundations

Detail plan for the first implementation phase. After Phase 1 the API
supports the new settings, all the new `order_by` keys, and emits a
`BrowserTableRowSerializer` when the user's `view_mode` is `"table"`.
No frontend work yet.

Prerequisites: `00-plan.md` Decisions section is current.

## Corrections + key facts from Phase 1 research

These either correct the Phase 0 plan or sharpen it:

- **Comic has FK fields only** — no denormalized
  `publisher`/`series`/`volume` CharFields on Comic itself. The
  CharFields with those names live on `ComicFTS`
  ([comic.py:367](codex/models/comic.py:367), `managed = False`), an
  FTS5 virtual table used for search, *not* ordering.
- **`publisher_name`, `series_name`, `volume_name`, `volume_number_to`
  are already F-expression annotations** wired in
  [`GroupNameMixin.annotate_group_names`](codex/views/mixins.py:88).
  They're gated by `cls.TARGET in _GROUP_NAME_TARGETS`. Adding more
  FK-name annotations (`imprint_name`, `country_name`, etc.) is just
  more entries here.
- **`BROWSER_DEFAULTS`** in
  [codex/choices/browser.py:94](codex/choices/browser.py:94) is *not*
  the runtime source of truth — backend defaults come from Django
  model field metadata via
  [`get_browser_default_params()`](codex/views/settings.py:345).
  `BROWSER_DEFAULTS` is consumed by the frontend (via the generated
  `browser-choices.json`) and as documentation. Adding a new model
  field with a default propagates automatically; we update
  `BROWSER_DEFAULTS` for consistency and frontend init.
- **Migration convention**: autogenerated schema migrations are the
  norm; `RunPython` data migrations exist for backfills (e.g.
  [0039_metron_age_rating_default_view_and_performance.py](codex/migrations/0039_metron_age_rating_default_view_and_performance.py)).
  For Phase 1 we need only schema migrations — model defaults backfill
  cleanly.
- **`_ORDER_AGGREGATE_FUNCS`**
  ([annotate/order.py:32](codex/views/browser/annotate/order.py:32))
  maps order keys to aggregate functions (`Avg`, `Min`, `Max`) used on
  *group rows* to compute `order_value` from child comics. Comics
  themselves order by indexed fields directly via
  [`add_order_by`](codex/views/browser/order_by.py:58). Each new
  order key needs an entry in *both* paths.
- **No M2M aggregations exist in the browser pipeline today.** The
  closest analogue is `JsonGroupArray("id", distinct=True)` on
  Comic.ids ([annotate/order.py:281](codex/views/browser/annotate/order.py:281)).
  For Phase 1 we'll add `JsonGroupArray` (or `GroupConcat` from
  [models/functions.py:28](codex/models/functions.py:28)) for selected
  M2M columns — see "M2M strategy" below.
- **No browser ordering tests exist today.** We add one per new
  `order_by` key in Phase 1 (and ideally retroactively for the existing
  13, but that's a follow-up).

## Phase 1 design decisions to confirm

These are smaller than Phase 0 but real. Calling them out so we agree
before code goes in.

### D1. `view_mode` and `table_columns` live on `SettingsBrowser` directly

Both are user-level scalars (well, `table_columns` is a JSON map).
They follow the existing pattern of `top_group`, `custom_covers`, etc.
— direct fields on `SettingsBrowser`. No nested model, no separate
table.

```python
# codex/models/settings.py — additions to SettingsBrowser
view_mode = CharField(
    max_length=8,
    choices=BROWSER_VIEW_MODE_CHOICES,
    default="cover",
)
table_columns = JSONField(default=dict)   # {"p": ["cover", ...], ...}
```

`table_columns` defaults to `{}` (not the per-top-group default map),
so we can distinguish "user has not customized" from "user picked
empty" — when the value for a top-group is missing or empty list, the
backend computes the default from the column registry. Same pattern
as the column-visibility map in `00-plan.md`.

### D2. `columns=` query param up-front (decided)

Table-view browse requests carry a `columns=` query param listing the
column keys the frontend wants. Backend builds the queryset
annotations dynamically: scalar/FK joins added only when requested,
M2M aggregates added only for requested M2M columns. The
`table_columns` setting still persists the user's choice across
sessions; the frontend reads it and translates to `columns=` on each
request.

Cover view is untouched — same fixed `BrowserCardSerializer` shape, no
`columns=` involved.

Always-on response fields (regardless of `columns=`):
- `pk` — required for row keying.

Everything else — including `name`, `cover_pk`, `cover_custom_pk` — is
gated by being in the `columns=` set. The frontend's column picker
(Phase 4) will enforce sensible minimums on its own (e.g. forbid
removing both name and cover).

Validation:
- Every entry in `columns=` must be a registry key (400 otherwise).
- Sorting by a column not in `columns=` is *allowed* (cover view's
  order-by dropdown surfaces the full enum). In table-view UI you
  can only click visible headers, so practical UX prevents this.

### D3. `table_columns` JSON shape

```json
{
  "p": ["cover", "name", "child_count"],
  "c": ["cover", "name", "issue_number", "series_name",
        "volume_name", "year", "page_count", "size"]
}
```

Map keyed by single-char top-group. Missing keys = use defaults.
Empty list under a key = also use defaults (treat as "user reset");
this matches the model field's behavior on PATCH-with-`{}`.

Sibling field `table_cover_size` (CharField, choices `xs|sm`,
default `sm`) for the cover-size option from Phase 0.

### D4. Column registry exposed via `browser-choices.json`

The registry is static — defined in Python, doesn't depend on user
state. Bake it into the existing `browser-choices.json` build via
[`choices_to_json.py`](codex/choices/choices_to_json.py). Frontend
picks it up at app load, no new endpoint, no live fetch.

Trade-off: any registry change requires `make build-choices` and a
frontend rebuild. Same constraint as the existing choices, so no
regression in workflow.

- **M2M aggregation in columns= path**
  (`tests/test_browser_table_m2m.py`): requesting
  `columns=cover,name,genres` returns aggregated genre names per row;
  requesting only scalar columns produces a queryset with zero M2M
  annotations (assert via `queryset.query.annotations`).

## Risks + open items

- **FK-path ordering performance** — `order_by("series__name")` adds
  an INNER JOIN. Indexed, but on libraries with 100k+ comics the join
  cost may show up. Mitigation: profile with a representative
  fixture; if a specific FK is slow, fall back to the `_ORDER_AGGREGATE_FUNCS`
  path with `Min`/`Max` to push the work into a subquery.
- **NULL ordering** — Django defaults to NULLs-first ascending. For
  sparse FK-backed columns (`country`, `tagger`, etc.) NULLs-last is
  often what users want. Decision for v1: ship default behavior; add
  `nulls_last=True` selectively in a follow-up if it bites.
- **`issue_count` annotation** — doesn't exist today (only
  `child_count`). Phase 1 needs to add it. Cheap (`Count("comic")`
  with a filter), but it's net-new code in
  [annotate/order.py](codex/views/browser/annotate/order.py).
- **`bookmark_updated_at` ordering** has special-case logic in
  [order_by.py:54](codex/views/browser/order_by.py:54). Verify it
  still works after the enum expansion.
- **`age_rating` is already in `BROWSER_ORDER_BY_CHOICES`** — the
  registry should not double-register it. Plan accordingly.
