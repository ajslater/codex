# Phase 2 — Frontend Table Scaffold

First-pass UI for the table-view alternative. After Phase 2 a user can
toggle into table mode in the top toolbar and see comics rendered as
rows with the registry's default columns, sort by clicking column
headers, and toggle back to cover view.

Out of scope (separate phases per `00-plan.md`):
- Bulk-selection wiring inside the table — Phase 3.
- Column-picker dialog — Phase 4.
- Mobile auto-fallback — Phase 5.

## Existing frontend (recap)

- `frontend/src/browser.vue` — page shell (toolbar + main + nav).
- `frontend/src/components/browser/main.vue` — grid of `<BrowserCard>` from `state.page.groups + state.page.books`.
- `frontend/src/components/browser/toolbars/top/browser-toolbar-top.vue` — hosts `top-group-select`, `order-by-select`, `order-reverse-button`, `filter-by-select`, search.
- `frontend/src/stores/browser.js` — Pinia store. `state.settings` mirrors persisted settings; `loadBrowserPage` calls `API.getBrowserPage(route.params, this.settings, mtime, ...)`.
- `frontend/src/api/v3/browser.js` — `getBrowserPage` sends the whole settings object as query params via `serializeParams`.
- Generated choices: `frontend/src/choices/browser-defaults.json` already carries `viewMode`, `tableColumns`, `tableCoverSize`. `browser-table-columns.json` and `browser-table-default-columns.json` are now generated by `make build-choices`.

## Architecture

### Store: thread the new settings through

`state.settings` gains `viewMode`, `tableColumns`, `tableCoverSize` initialised from `BROWSER_DEFAULTS`. They round-trip through `loadSettings()` / `setSettings()` already (the backend exposes them — Phase 1 Step 2).

`loadBrowserPage` injects a `columns` query param when `viewMode === "table"`:

```js
const params = { ...this.settings };
if (params.viewMode === "table") {
  const cols = params.tableColumns?.[params.topGroup]
            ?? defaultColumnsFor(params.topGroup);
  params.columns = cols.join(",");
}
return API.getBrowserPage(route.params, params, mtime, ...);
```

`defaultColumnsFor(topGroup)` reads `browser-table-default-columns.json`.

The response branch: when the body has `rows`, it's a table response;
when it has `groups`/`books`, it's a card response. Store keeps them
both on `state.page`; the renderer picks whichever is present.

### Renderer: switch in `main.vue`

```vue
<BrowserTable v-if="settings.viewMode === 'table'" />
<v-pull-to-refresh v-else>...<BrowserCard /></v-pull-to-refresh>
```

### `BrowserTable.vue`

Vuetify's `v-table` (not `v-data-table-server`) — the existing nav
toolbar already drives URL-based pagination, and the rows come pre-
paginated from the server. Reusing the existing nav avoids reimplementing
page navigation inside the data table.

Structure:
```vue
<v-table fixed-header>
  <thead><tr>
    <th v-for="col in visibleColumns"
        :class="{ sortable: isSortable(col) }"
        @click="onHeaderClick(col)">
      {{ labelFor(col) }}
      <v-icon v-if="orderBy === sortKeyFor(col)">
        {{ orderReverse ? 'mdi-arrow-down' : 'mdi-arrow-up' }}
      </v-icon>
    </th>
  </tr></thead>
  <tbody>
    <tr v-for="row in rows" :key="row.pk" @click="open(row)">
      <td v-for="col in visibleColumns">
        <BrowserTableCell :column="col" :row="row" :cover-size="tableCoverSize" />
      </td>
    </tr>
  </tbody>
</v-table>
```

`BrowserTableCell` for v1: small switch on column type. Cover renders
an `<img>` sized by `tableCoverSize`; lists (M2M arrays) join with `,`;
booleans render `Yes`/`No`/empty; everything else stringifies.

### View-mode toggle

`browser/toolbars/top/view-mode-toggle.vue`: single icon button that
flips `settings.viewMode`. Renders next to the existing settings
button. `mdi-view-grid` ↔ `mdi-table`.

### Hiding order controls in table mode

Wrap `order-by-select` and `order-reverse-button` in `v-if` on
`viewMode !== 'table'`. The header-click handler in `BrowserTable`
takes over the same `orderBy`/`orderReverse` plumbing.

### Header-click sort

```js
onHeaderClick(col) {
  const sortKey = sortKeyFor(col);
  if (!sortKey) return;
  if (this.orderBy === sortKey) {
    this.setSettings({ orderReverse: !this.orderReverse });
  } else {
    this.setSettings({ orderBy: sortKey, orderReverse: false });
  }
}
```

## Open / deferred

- **Cell renderers**: v1 uses string fallback for most types. Per-type components (`cell-decimal`, `cell-date`, etc.) are an iterative polish later in this phase or in Phase 6.
- **`order_value` column**: cover view sometimes displays an order-by caption based on `order_value`. Table view doesn't need it; the column header itself communicates the active sort. Skipped.
- **Column picker dialog**: Phase 4. v1 ships with the registry defaults only; the user can still see the table, just can't customize columns yet (other than via PATCH to the API directly).
